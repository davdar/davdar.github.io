# Type Systems

Type systems are common to many, but not all programming languages. C, C++,
Java and Haskell are examples of languages with type systems, and Python and
Javascript are examples of languages without type systems.

Type systems can serve a number of purposes:

- Necessary information for the compiler
- Checking for programming errors

Type systems are often *overly restrictive*, meaning there are programs which
don't have errors that will not pass the type checker. Type systems are also
*fully automated bug finders*, so they can be immensely valuable as well. You
have spent nearly a whole semester now arguing with Haskell's type checker, so
you are at least familiar with the user-interaction component of type systems
and type checkers.

# Inductively Defined Sets

In this class you've seen the natural numbers defined like so:

    в„• вүң {0,1,вҖҰ}

This definition of the natural numbers is вҖңinformalвҖқ. How do we know what
really comes after the `вҖҰ`? A more pedantic definition of the natural numbers
would be as an inductively constructed set, using the same notation we have
already used to construct syntax. Just like we have already seen:

    e вҲҲ expr в©ҙ i | e + e

we could define the natural numbers as:

    n вҲҲ в„• в©ҙ 0 | succ n

Here `succ` isn't an operation, it's a piece of uninterpreted syntax. The
number three is then a linked list:

    three вҲҲ в„• вүң succ (succ (succ 0))

Let's write the natural numbers one last time using a new type of funny
notation:

    в”Ңв”Җв”Җв”Җв”Җв”Җв”җ
    в”Ӯn вҲҲ в„•в”Ӯ
    в””в”Җв”Җв”Җв”Җв”Җв”ҳ

    в”Җв”Җв”Җв”Җв”Җ (1)
    0 вҲҲ в„•

    n вҲҲ в„•
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ (2)
    succ n вҲҲ в„•

The first rules reads: вҖңzero is a natural number.вҖқ The second rule reads вҖң*if*
`n` is a natural number, *then* `succ n` is a natural numberвҖқ. These are called
inference rules, and the general idea for reading them is вҖңif all of the stuff
above the line is true, then the thing below the line is trueвҖқ. Together these
two rules (1вҖ“2) are equivalent to the writing:

    n вҲҲ в„• в©ҙ 0 | succ n

So why would we use вҖңinference rulesвҖқ? They appear for now just a more verbose
way of defining tree-like sets. Let's do a quick case study on trees.

## Well-balanced Trees

Let's define (inductively) the set of trees:

    t вҲҲ tree в©ҙ L | вҹЁt,tвҹ©

we are interested in *well-balanced* trees; for example this tree is
well-balanced:

    вҹЁвҹЁL,Lвҹ©,вҹЁL,Lвҹ©вҹ©

and these trees are not:

    вҹЁвҹЁL,Lвҹ©,Lвҹ©

    вҹЁL,вҹЁL,Lвҹ©вҹ©

next we create an *inductively defined relation* using the inference rules
described above, written `t в–і n` which is read вҖңtree `t` is well balanced at
height `n`вҖқ:

    в”Ңв”Җв”Җв”Җв”Җв”Җв”җ
    в”Ӯt в–і nв”Ӯ
    в””в”Җв”Җв”Җв”Җв”Җв”ҳ
   
    в”Җв”Җв”Җв”Җв”Җ (leaf)
    L в–і 1

    tвӮҒ в–і n    tвӮӮ в–і n
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ (node)
    вҹЁtвӮҒ,tвӮӮвҹ© в–і (n + 1)


it is the case that if `t в–і n` then `t` is well-balanced. We can write a
*proof* of this fact directly using inference rules. E.g., to prove that this
tree is well-balanced:

    вҹЁвҹЁL,Lвҹ©,вҹЁL,Lвҹ©вҹ©

we can write this proof that it well-balanced at height 3:

    в”Җв”Җв”Җв”Җв”Җ    в”Җв”Җв”Җв”Җв”Җ    в”Җв”Җв”Җв”Җв”Җ    в”Җв”Җв”Җв”Җв”Җ  (leaf Г— 4)
    L в–і 1    L в–і 1    L в–і 1    L в–і 1
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ  (node Г— 2)
      вҹЁL,Lвҹ© в–і 2         вҹЁL,Lвҹ© в–і 2
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ (node)
          вҹЁвҹЁL,Lвҹ©,вҹЁL,Lвҹ©вҹ© в–і 3

this is the primary reason why inference rules are written in this funny wayвҖ“so
we can stack them together to create a *proof tree* of some inductively defined
property.

# Scoping

Let's define an inductive relation for well-scoping for a tiny programming
language. The relation is `S вҠў e` and is pronounced вҖңin a scope with `S` bound
variable, program fragment `e` is well-scopedвҖқ:

    x вҲҲ var вүҲ symbol
    e вҲҲ expr в©ҙ i | e + e
             | x | let x = e in e

    S вҲҲ scope вүң в„ҳ(var)

    в”Ңв”Җв”Җв”Җв”Җв”Җв”җ
    в”ӮS вҠў eв”Ӯ
    в””в”Җв”Җв”Җв”Җв”Җв”ҳ
    
    в”Җв”Җв”Җв”Җв”Җ (int)
    S вҠў i

    S вҠў eвӮҒ
    S вҠў eвӮӮ
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ (plus)
    S вҠў eвӮҒ + eвӮӮ

    x вҲҲ S
    в”Җв”Җв”Җв”Җв”Җ (var)
    S вҠў x

    S вҠў eвӮҒ
    {x}вҲӘS вҠў eвӮӮ
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ (let)
    S вҠў let x = eвӮҒ in eвӮӮ

And we have a pair of theorems:

- If `S вҠў e` then `FV(e) вҠҶ S`
- If `FV(e) вҠҶ S` then `S вҠў e`

and a pair of corollaries:

- If `вҲ… вҠў e` then `e` is closed.
- If `e` is closed then `вҲ… вҠў e`.

(The proofs are by structural induction on the term `e`вҖ”we won't get into the
proof details in this class.)

# Simple Types (no variables)

Let's start with a simple type system for a programming language with just
integers, plus, booleans, and if:

    e вҲҲ expr в©ҙ i | e + e
             | b | if e then e else e

    П„ вҲҲ type в©ҙ int | bool

a *simple type system* for this language relates a term to a type, written `e :
П„` and pronounced вҖңprogram fragment `e` has type `П„`вҖқ. The type system is
defined inductively (using inference rules) and is designed such that if `e :
П„`, then `e` is guaranteed to (1) not crash the interpreter due to a scope or
runtime type error, and (2) return a value of type `П„` as the final result.

    в”Ңв”Җв”Җв”Җв”Җв”Җв”җ
    в”Ӯe : П„в”Ӯ
    в””в”Җв”Җв”Җв”Җв”Җв”ҳ
    
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җ (int)
    i : int

    eвӮҒ : int
    eвӮӮ : int
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ (plus)
    eвӮҒ + eвӮӮ : int

    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ (bool)
    b : bool

    eвӮҒ : bool
    eвӮӮ : П„
    eвӮғ : П„
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ (if)
    if eвӮҒ then eвӮӮ else eвӮғ : П„

The (if) rule is worth discussing. First, the use of the same `П„` above the
line can be read to say вҖңwhen `eвӮӮ` and `eвӮғ` have the same type, and that type
is `П„`, then the resulting type of the `if` expression is that same `П„`вҖқ. That
is, it doesn't matter what type the result of the `if` expression has, so long
as it is the same as the sub-terms `eвӮӮ` and `eвӮғ`.

## Simple Types (variables)

Let's add variables and let to our language:

    e вҲҲ expr в©ҙ i | e + e
             | b | if e then e else e
             | x | let x = e in e

    П„ вҲҲ type в©ҙ int | bool
    
    О“ вҲҲ tenv вүң var вҮҖ type

    в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
    в”ӮО“ вҠў e : П„в”Ӯ
    в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
    
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ (int)
    О“ вҠў i : int

    О“ вҠў eвӮҒ : int
    О“ вҠў eвӮӮ : int
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ (plus)
    О“ вҠў eвӮҒ + eвӮӮ : int

    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ (bool)
    О“ вҠў b : bool

    О“ вҠў eвӮҒ : bool
    О“ вҠў eвӮӮ : П„
    О“ вҠў eвӮғ : П„
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ (if)
    О“ вҠў if eвӮҒ then eвӮӮ else eвӮғ : П„

    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ (var)
    О“ вҠў x : О“(x)

    О“ вҠў eвӮҒ : П„вӮҒ
    О“[xвҶҰП„вӮҒ] вҠў eвӮӮ : П„вӮӮ 
    в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ (let)
    О“ вҠў let x = eвӮҒ in eвӮӮ : П„вӮӮ

Notice the new structure `О“`, which acts like the scope environment `S` from
before, and is a static approximation of the interpreter environment `Оі`:
whereas `Оі` maps variables to values, `О“` maps variables to their types.

The type environment `О“` serves to place *assumptions* on the types of free
variables in the term `e`. A top-level program will have no free variables, and
a typing judgment for a top-level term `e` will look like `вҲ… вҠў e : П„`. Because
this is so commonвҖ”notating a type judgment with an empty type environment, we
just write `вҠў e : П„` when the type environment is empty.

## Specification vs Algorithm

The relation `О“ вҠў e : П„` defined above is an *inductively defined relation*
which tells you that a good thing is true about `e` when `О“ вҠў e : П„`. However,
we are interested in an *algorithm* that *checks* whether or not `О“ вҠў e : П„`
for some `О“` and `П„`. This algorithm is called a *type checker*, whereas the
inductively defined relation is called a *type system*. The type checker is (1)
algorithmic, and (2) should return results consistent with the definition of
the type system (the specification).

Here is the type checker written in algorithm form, as in interpreter:

    вҰҮ_вҰҲ вҲҲ expr вҶ’ tenv вҶ’ maybe(type)
    вҰҮiвҰҲ(О“) вүң int
    вҰҮeвӮҒ + eвӮӮвҰҲ(О“) вүң int
      where int = вҰҮeвӮҒвҰҲ(О“)
            int = вҰҮeвӮӮвҰҲ(О“)
    вҰҮbвҰҲ(О“) вүң bool
    вҰҮif eвӮҒ then eвӮӮ else eвӮғвҰҲ(О“) вүң П„
      where bool = вҰҮeвӮҒвҰҲ(О“)
            П„    = вҰҮeвӮӮвҰҲ(О“)
            П„    = вҰҮeвӮғвҰҲ(О“)
    вҰҮxвҰҲ(О“) вүң П„
      where П„ = О“(x)
    вҰҮlet x = eвӮҒ in eвӮӮвҰҲ(О“) вүң П„вӮӮ
      where П„вӮҒ = вҰҮeвӮҒвҰҲ(О“)
            П„вӮӮ = вҰҮeвӮӮвҰҲ(О“[xвҶҰП„вӮҒ])

Notice the following:

- The type checker `вҰҮ_вҰҲ` has very similar structure as the normal interpreter
  `вҹҰ_вҹ§`.
- In the `if` case, there are two clauses for `П„ = вҖҰ`. This means that *both*
  of those `П„` have to be the same, so in the full implementation of the type
  checker, there will be an equality check that the type result from `вҰҮeвӮӮвҰҲ` and
  `вҰҮeвӮғвҰҲ` are equal.
- The let rule for `вҰҮ_вҰҲ` looks just like the one for `вҹҰ_вҹ§`, but with types
  instead of values.
